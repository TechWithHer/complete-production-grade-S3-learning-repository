# Scenario

CloudMart has been operating its Amazon S3 platform for over a year.

The platform now stores thousands of files generated by different departments.

```
documents/
images/
logs/
backups/

finance/
```

Initially, every uploaded file remained in Amazon S3 forever.

After several months, the operations team notices several problems:

* Old application logs are consuming storage.
* Temporary files uploaded by users are never deleted.
* Previous object versions continue growing because Versioning is enabled.

The company realizes that manually managing storage is impossible.

As the Cloud Engineer, your responsibility is to automate the entire lifecycle of CloudMart's data.

Welcome to **Amazon S3 Lifecycle Management**.

# Why Lifecycle Rules Exist

Imagine you own a warehouse.

Every day:

* New products arrive.
* Old products become obsolete.
* Temporary packing materials accumulate.
* Damaged items must be discarded.
* Archived records must be retained for legal purposes.

If no cleanup process exists,

the warehouse eventually becomes full, expensive, and difficult to manage.

Amazon S3 faces the same challenge.

Without automation:

* Storage grows continuously.
* Costs increase.
* Old versions accumulate.
* Temporary files remain forever.
* Compliance becomes difficult.

Lifecycle Rules solve this problem automatically.

---

# What Is a Lifecycle Rule?

A Lifecycle Rule is an automated policy that instructs Amazon S3 to perform specific actions on objects after a defined period.

Amazon S3 evaluates these rules in the background without requiring any scripts, cron jobs, or manual intervention.

Think of a Lifecycle Rule as an automated housekeeping service.

---

# Lifecycle Actions

A Lifecycle Rule can perform several actions.

| Action                        | Purpose                                   |
| ----------------------------- | ----------------------------------------- |
| Transition Objects            | Move objects to another Storage Class     |
| Expire Objects                | Permanently delete current objects        |
| Delete Non-current Versions   | Remove old versions created by Versioning |
| Delete Expired Delete Markers | Clean up delete markers                   |
| Abort Multipart Uploads       | Remove abandoned uploads                  |

Each action solves a different operational problem.

---

# Understanding Object Age

Lifecycle Rules operate using the age of an object.

Example:

| Day     | Event                    |
| ------- | ------------------------ |
| Day 1   | Object Uploaded          |
| Day 30  | Transition Rule Executes |
| Day 90  | Archive Rule Executes    |
| Day 365 | Expiration Rule Executes |

Amazon S3 calculates object age automatically.

You never manually execute Lifecycle Rules.

---

# Lifecycle Rule Components

Every Lifecycle Rule contains four major components.

## 1. Rule Name

Example:

```
delete-temp-files
```

Always use descriptive names.

Production environments often contain dozens of lifecycle policies.

---

## 2. Scope

A rule can apply to:

Entire Bucket

OR

Specific objects.

Filtering options include:

* Prefix
* Object Tags
* Object Size

---

## 3. Action

Examples:

* Transition
* Expiration
* Delete Marker Cleanup
* Abort Multipart Upload

---

## 4. Timing

Example:

```
Delete after 30 days
```

or

```
Delete previous versions after 90 days
```

---

# Lifecycle Evaluation

One common misconception is:

> "At exactly midnight on Day 30 my object will disappear."

Not necessarily.

Lifecycle processing is asynchronous.

Amazon S3 periodically evaluates objects and executes eligible actions.

The action may occur slightly after the configured age.

This is expected behavior.

# Common Mistakes

* Creating overly broad rules that affect the entire bucket unintentionally.
* Assuming Lifecycle Rules execute immediately after the configured age.
* Forgetting to test rules on non-production data first.
* Using unclear rule names.
* Applying the wrong prefix filter.

---

# Troubleshooting

### Problem

My objects were not deleted after 30 days.

Possible Cause:

Lifecycle processing is asynchronous.

Allow additional time for Amazon S3 to process eligible objects.

---

### Problem

Objects inside `documents/` were not deleted.

Possible Cause:

The Lifecycle Rule applies only to the `temp/` prefix.

Verify your filter configuration.

---

### Problem

I accidentally configured the wrong expiration period.

Possible Cause:

Review and edit the Lifecycle Rule before it affects production data.

# Understanding Versioning and Lifecycle

Recall from Chapter 5 that Versioning changes how Amazon S3 handles updates and deletions.

Without Versioning:

```text
report.pdf
```

Deleting the object permanently removes it.

With Versioning enabled:

```text
report.pdf
        │
        ├── Version 1
        ├── Version 2
        ├── Version 3
        └── Delete Marker
```

Deleting the object creates a **Delete Marker**, while older versions continue consuming storage.

Lifecycle Rules can automatically clean up these old versions.

This is one of the most common production configurations.

---

# Current Version vs Non-Current Version

Understanding this distinction is essential.

| Type                | Description                              |
| ------------------- | ---------------------------------------- |
| Current Version     | Latest version of an object              |
| Non-Current Version | Previous versions retained by Versioning |

Example:

```text
employee.xlsx

Current Version
Version ID: V5

Older Versions
V1
V2
V3
V4
```

Although users normally see only V5, the earlier versions continue occupying storage until they are explicitly deleted or removed by a Lifecycle Rule.

---

# Understanding Delete Markers

Delete Markers exist only in versioned buckets.

Suppose you delete:

```text
employee-policy.pdf
```

Amazon S3 actually creates:

```text
Delete Marker
```

The previous object versions still exist.

This allows the object to be restored.

However,

over time,

Delete Markers also accumulate.

Large production buckets may contain millions of unnecessary Delete Markers.

Lifecycle Rules can automatically remove them.

---

Observe the available configuration.

Notice that this action specifically targets Delete Markers rather than the object itself.

# Understanding Multipart Uploads

Multipart Upload allows very large files to be uploaded in smaller pieces.

Instead of uploading a 20 GB file in one request:

```text
20 GB
```

Amazon S3 divides the upload into multiple parts.

```text
Part 1
Part 2
Part 3
...
Part N
```

After all parts arrive,

Amazon S3 combines them into a single object.

---

# Why Multipart Upload Cleanup Matters

Imagine a user begins uploading:

```text
database-backup.tar
```

The upload fails halfway through because:

* Internet connection drops.
* Laptop shuts down.
* Application crashes.

The uploaded parts continue consuming storage even though no usable object exists.

Thousands of failed uploads can waste significant storage.

Lifecycle Rules solve this automatically.

---

# Understanding Object Tags

Until now, every Lifecycle Rule has used **Prefix Filters**.

Amazon S3 also supports **Object Tags**.

Example:

| Object       | Tag                    |
| ------------ | ---------------------- |
| payroll.xlsx | Department = HR        |
| invoice.pdf  | Department = Finance   |
| logo.png     | Department = Marketing |

Unlike prefixes, tags describe business metadata rather than object location.

---

# Prefix vs Tag Filters

| Prefix                       | Tags                               |
| ---------------------------- | ---------------------------------- |
| Folder-based                 | Metadata-based                     |
| Easier to understand         | More flexible                      |
| Limited to object path       | Can represent business information |
| Common in small environments | Common in enterprise environments  |

Large organizations often rely heavily on object tags.

---

# Rule Evaluation

What happens if an object matches multiple Lifecycle Rules?

Example:

```text
documents/payroll.xlsx
```

Matches:

Rule 1

```text
documents/
```

Rule 2

```text
Department = HR
```

Amazon S3 evaluates all applicable rules.

The service resolves conflicts using documented lifecycle evaluation logic to determine which action should occur. When designing production environments, avoid creating overlapping or contradictory rules whenever possible.

Rather than manually managing storage,

# Best Practices

For production environments:

* Use descriptive Lifecycle Rule names.
* Test rules in development before production.
* Keep Versioning enabled.
* Regularly review storage growth.
* Use Prefix Filters for simple environments.
* Use Object Tags for enterprise-scale governance.
* Enable Multipart Upload cleanup on every production bucket.
* Document business retention requirements before implementing technical policies.
* Review Lifecycle Rules annually as business requirements evolve.

---

# Common Mistakes

* Deleting compliance data too early.
* Forgetting that Versioning creates non-current objects.
* Ignoring Delete Markers.
* Never cleaning up incomplete multipart uploads.
* Creating overlapping Lifecycle Rules.
* Assuming Lifecycle actions occur immediately at midnight.

# Troubleshooting

### Problem

Previous versions are still consuming storage.

Possible Cause:

No rule exists for deleting non-current versions.

---

### Problem

Storage continues growing even after deleting objects.

Possible Cause:

Delete Markers and previous versions still exist.

---

### Problem

Lifecycle Rule is not affecting an object.

Possible Cause:

Verify:

* Prefix Filter
* Tag Filter
* Rule Status
* Object Age

---

### Problem

Multipart uploads are consuming storage.

Possible Cause:

Incomplete Multipart Upload cleanup has not been configured.

---

### Problem

Objects were deleted unexpectedly.

Possible Cause:

Review every Lifecycle Rule for overlapping filters and incorrect retention periods.

Always test Lifecycle Rules in a non-production environment before applying them to critical data.
